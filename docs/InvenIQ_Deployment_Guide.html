1 How the Module Access System Works

InvenIQ uses a two-account, three-layer access control system. You control exactly which modules a client can see and use — enforced at both the UI and API level.

Two Accounts

Account Set in .env Purpose Access
Client account AUTH_USERNAME / AUTH_PASSWORD What the client logs in with every day Only the modules you list in AUTH_ALLOWED_MODULES
Owner/Developer OWNER_USERNAME / OWNER_PASSWORD Your private backdoor admin account Full access to all 26 modules, always
Important
Leave OWNER_USERNAME blank to disable the backdoor account entirely on installations where you don't need it.

Three Enforcement Layers

LayerWhereWhat it does
1 — Sidebar (UI) Frontend / React Navigation menu only shows allowed modules. Restricted items are completely hidden — not greyed out, just gone.
2 — Keyboard shortcuts (UI) Frontend / React g → letter navigation is filtered. Pressing a shortcut for a restricted module does nothing.
3 — Backend API (hard wall) Backend / FastAPI Every API call is verified against the JWT token's allowed_modules claim. Blocked calls return 403. No data is returned regardless of what the UI does.

How the JWT Token Works

On login, the server issues a JWT (JSON Web Token) containing the user's role and allowed_modules. Every API call the browser makes includes this token. The backend's ModuleAccessMiddleware reads the token on every request — it never queries the database again for permissions.

Middleware Execution Order
AuthMiddleware (validates JWT, sets user) → ModuleAccessMiddleware (checks allowed modules) → GZip → CORS → Routes

Complete Login Flow for a Restricted Client

1
Client enters username + passwordPOST /api/auth/login
2
Server validates credentialsChecks DB → owner account → client account (priority order)
3
Server issues JWT containing:role=client, allowed_modules=quotes,customers,catalog,chatbot,settings,about
4

5

2 .env Configuration Reference

All client-specific settings are in backend/.env. Never commit this file to Git.

# ── OpenAI (required for AI chat features) ───────────────────────────
OPENAI_API_KEY=sk-...your-key-here...

# ── MySQL Database (optional — leave blank for demo mode) ─────────────
MYSQL_HOST=
MYSQL_PORT=3306
MYSQL_USER=stocksense
MYSQL_PASSWORD=your_db_password
MYSQL_DB=stocksense_inventory

# ── Security ──────────────────────────────────────────────────────────
# Generate: python -c "import secrets; print(secrets.token_hex(32))"
JWT_SECRET_KEY=CHANGE-THIS-TO-UNIQUE-32-CHAR-STRING
ACCESS_TOKEN_EXPIRE_HOURS=8

# ── Client login account ──────────────────────────────────────────────
AUTH_USERNAME=client
AUTH_PASSWORD=strong_client_password
AUTH_DISPLAY_NAME=Client Company Name
AUTH_EMAIL=client@company.com
AUTH_ROLE=client
AUTH_ALLOWED_MODULES=quotes,customers,catalog,chatbot,settings,about

# ── Developer/owner backdoor (private — never share this) ─────────────
OWNER_USERNAME=devadmin
OWNER_PASSWORD=your_private_admin_password
OWNER_DISPLAY_NAME=Developer
OWNER_EMAIL=dev@yourcompany.com

# ── CORS ──────────────────────────────────────────────────────────────
ALLOWED_ORIGINS=http://localhost:8000,http://127.0.0.1:8000

3 Source Code Protection Options

OptionProtectionClient runs offline?Effort
Host on your server (SaaS) MAXIMUM — zero files leave your hands No — needs internet Medium (one-time VPS setup)
PyInstaller .exe package HIGH — no .py files, compiled binary Yes — fully offline Low (run one script)
Docker image HIGH — source inside image layer Yes — needs Docker Desktop Medium
Copy full folder (current) NONE — full source exposed Yes Zero
Recommended: PyInstaller Package (like InvenIQ2)
Client receives a ZIP containing only: InvenIQ-server.exe + web/ + .env + START-SERVER.bat. No Python source. No JSX source. Double-click to run.
Note on the React frontend
npm run build already produces minified, mangled JavaScript. Your JSX/React source is not readable in the build output. The frontend is already partially protected as long as you only give clients the frontend/build/ folder, not frontend/src/.

4 Client Package System (PyInstaller)

Based on the InvenIQ2 project. Four files need to be created in InvenIQ to enable this system:

Files to Create

FileLocationPurpose
run.pybackend/run.pyPyInstaller entry point — starts uvicorn, loads .env from beside the .exe
InvenIQ.specbackend/InvenIQ.specPyInstaller config — lists all routers, hidden imports, compiles to InvenIQ-server.exe
create-client-package.batproject rootMaster build script — builds React + exe + assembles + ZIPs in one command
.env.templateproject rootBlank .env that goes inside the client ZIP (includes AUTH_* and OWNER_* fields)

How to Build a Client Package (once files exist)

create-client-package.bat "ClientName"

This runs 4 automatic steps:

1
React build — npm run build with GENERATE_SOURCEMAP=false
2
Python compile — PyInstaller bundles entire backend into InvenIQ-server.exe
3
Package assembly — clean folder with no source files, only the built artifacts
4
ZIP creation — InvenIQ-Client-ClientName.zip ready to send

What the Client ZIP Contains

InvenIQ-Client-ClientName.zip
  InvenIQ-server.exe  ← compiled backend — no .py files
  web\                ← compiled React — no .jsx source
  .env                ← client fills in OpenAI key + credentials
  START-SERVER.bat   ← double-click to run
  README.txt         ← 5-line instructions

5 Deploying to a Client's Windows Machine

Method A — Full Source (fastest, source exposed)

Copy the entire c:\InvenIQ folder to the client machine. Client runs two scripts:

1
Prerequisites — Install Python 3.10+ and Node.js 18+ LTS on the client machine. Check "Add Python to PATH" during Python install.
2
Create backend/.env — Copy deploy\client.env.example to backend\.env. Fill in OPENAI_API_KEY, AUTH_USERNAME/PASSWORD, AUTH_ROLE=client, AUTH_ALLOWED_MODULES, OWNER_USERNAME/PASSWORD, JWT_SECRET_KEY.
3
First-time setup — Double-click deploy\windows\install.bat. Installs Python packages, builds React. Takes ~2 minutes.
4
Daily startup — Double-click start_prod.bat. Opens browser at http://localhost:8000 automatically.

Method B — PyInstaller Package (source protected)

Client receives only the ZIP file. No Python or Node.js needed on their machine.

1
You build the package (on your machine) — Run create-client-package.bat "ClientName". Takes ~5 minutes. Produces InvenIQ-Client-ClientName.zip.
2
Edit .env inside the ZIP — Before sending, open the ZIP, edit .env, fill in all client-specific credentials and module restrictions.
3
Send ZIP to client — Email, USB, file share, Google Drive — any method.
4
Client extracts ZIP — Then double-clicks START-SERVER.bat. Opens browser at http://localhost:8000.
5
Daily startup — Client double-clicks START-SERVER.bat every time they start work.

Method C — Hosted on Your Server (maximum protection)

1
Rent a VPS — DigitalOcean, Hostinger, AWS Lightsail etc. Ubuntu 22.04, ~$6–12/month.
2
Upload InvenIQ to server — SSH in, copy the project, run pip install -r requirements.txt, npm run build.
3
Create backend/.env on server — Same as Method A but with ALLOWED_ORIGINS=https://yourdomain.com.
4
Start server permanently — Use pm2 or systemd to keep it running 24/7.
5
Give client the URL — e.g. http://yourserver-ip:8000 or set up a domain. Client opens in any browser on any device. No files, no installation.

Client must run offline, source protection not priority Method A — Copy folder + install.bat Ready now
Multiple clients, clean separation per client Method C (separate .env per client) or Method B (separate ZIP per client) Needs setup
To create the PyInstaller package system (Method B)
Four files need to be created: backend/run.py, backend/InvenIQ.spec, create-client-package.bat, and .env.template. These are already built and tested in the InvenIQ2 project — they just need to be adapted for InvenIQ's additional routers and auth dependencies.
